REQUIRED WORKFLOW
Stage 0: Resolve label and bind context
- Resolve `\label{<PROPOSITION_LABEL>}` uniquely.
- Identify the containing file, statement text, and nearest deterministic insertion point for an include.
- Determine whether a proof for this statement exists, and where it is located.

Stage 1: Evidence capture
- Collect exact evidence anchors used for the review:
  - file paths
  - environment boundaries (statement/proof blocks)
  - referenced local lemmas/theorems/definitions explicitly used by the proof
- Keep `observed facts` separate from `inferred implications`.

Stage 2: Deep proof review (mandatory deep pass)
- If proof exists, perform a full logical audit:
  - assumption sufficiency
  - inference validity
  - hidden steps and unjustified jumps
  - circular dependencies
  - scope/quantifier/domain consistency
  - edge-case stress checks implied by the statement assumptions
- Classify each detected issue by severity: `Critical`, `Major`, or `Minor`.
- If proof does not exist, skip logical audit and continue with explicit `No proof found` outcome.

Stage 3: Verdict synthesis
- Choose exactly one outcome:
  - `Sound` (no substantive logical defects found)
  - `Issues found` (one or more logical/material defects found)
  - `No proof found`
  - `Blocked` (insufficient evidence to complete reliable review)
- Provide short, precise justification tied to evidence anchors.

Stage 4: Write report file
- Report path:
  - `paper/reports/proof_review_<sanitized_label>.tex`
  - sanitize label by replacing non-alphanumeric characters with `_`.
- Report must contain:
  1) `\section{Proof Review: <PROPOSITION_LABEL>}`
  2) Target statement restatement and location
  3) Proof presence status and proof location (or explicit absence)
  4) Review methodology summary (what was checked)
  5) Final outcome: `Sound`, `Issues found`, `No proof found`, or `Blocked`
  6) Detailed findings list with severity labels and evidence anchors
  7) If `Issues found`: exact failing steps and why they fail
  8) If `Blocked`: exact missing evidence and what is needed
  9) End with `\clearpage`

Stage 5: Insert report include (minimal integration)
- Insert one include in the host manuscript file at the deterministic point nearest the reviewed statement/proof:
  - `\input{paper/reports/proof_review_<sanitized_label>}`
- Do not insert duplicates if an include for this label already exists.
- Do not make any other edits in the manuscript.

Stage 6: Build verification (mandatory)
- Run a manuscript build command and verify successful PDF generation after edits.
- Prefer repository-standard command:
  - from `paper/`: `latexmk -pdf -interaction=nonstopmode -file-line-error main.tex`
- If build fails, do not declare success; either fix only report/include-caused issues and rebuild, or report `Blocked` with exact failure details.
